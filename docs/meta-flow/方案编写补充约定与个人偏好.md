# 方案编写补充约定与个人偏好

> 说明：
>
> 1. 仓库级共享规范以 `AGENTS.md` 与 `delivery/rules/AGENTS.md` 为准；
> 2. 本文用于补充那些**不适合直接写入共享规范**、但在实际方案写作和修订中应优先参考的约定；
> 3. 若本文中的某条偏好后续被证明已稳定、可团队共享，再提升到仓库级规范。

---

## 1. 已经写入共享规范的内容

以下内容已经正式进入仓库共享规范，不再只是偏好：

1. 先核对事实，再写方案；
2. 优先最简方案；
3. 已废弃内容要彻底删除；
4. 阻塞问题、遗留问题和开放问题必须状态化；
5. 主选方案与备选方案并存，并写清切换条件；
6. 问题描述必须完整；
7. 目录设计要分层；
8. 方案收敛优先；
9. 精确匹配优先。

---

## 2. 本文记录的补充偏好

以下内容**暂不直接作为团队强制规范**，但在你当前的方案写作习惯中应优先遵守。

### 2.1 优先反映“当前真实状态”

写方案时优先回答：

1. 当前仓库真实实现是什么；
2. 当前目录和路径是否真的存在；
3. 官方文档是否支持当前写法；
4. 旧方案里的设计是否已经失效。

如果这几项没有先搞清楚，不应直接扩写新方案。

### 2.2 优先内联，不优先建设共享 template

除非已经明确存在稳定 owner 与长期复用价值，否则：

- 优先把结构要求内联到规则、Skill 或 Agent 文本中；
- 不优先新增共享 template 目录；
- 不为了“未来可能复用”提前建设复杂模板体系。

### 2.3 默认围绕目标项目根目录设计输出

当方案涉及“运行输出、过程文档、自动检查结果、人工检查点、交付物”时，默认应围绕：

- 当前被 Codex / Claude Code 打开的目标项目根目录；
- 而不是 meta 仓库自身的历史 `.output/` / `~/.meta-flow/` 目录。

这条偏好尤其适用于：

- 工作流输出位置设计；
- 过程文档与交付文档分层；
- 路径迁移和整改计划。

### 2.4 非托管边界必须写清

对于以下对象，要优先回答“是否托管”：

- `model`
- `provider`
- `mcp`
- `tools`
- 用户状态文件
- 用户本地 override / local 层

若不托管，应在方案中明确写“不托管”，而不是只写“后续再讨论”。

### 2.5 人工动作默认由用户自己执行

方案应清楚区分：

1. 系统自动完成的动作；
2. 需要提示用户但由用户自己完成的动作。

例如：

- 重启 Codex / Claude Code；
- 手工测试渲染后的效果；
- 手工执行 git 操作；
- 手工确认权限或可信路径。

### 2.6 权限与能力按需加载

默认不预加载高权限能力。

只有当用户真正调用对应能力、且权限不足时，才提示用户加载或授权。  
这条偏好适用于：

- `meta-improve`
- 跨仓库写入
- 需要可信目录 / workspace roots 的场景

### 2.7 exact 优先于模糊语义

若方案里涉及：

- 对象定位
- 规则匹配
- 版本对齐
- 平台路径判定

默认优先 exact 语义。  
只有在 exact 明显妨碍可用性、且问题已被明确识别时，才考虑放宽。

### 2.8 方案要形成“修改 -> 渲染 -> 日志 -> 测试提示”闭环

你当前偏好的完整链路是：

1. 修改 canonical source；
2. 记录本次改动；
3. 及时执行渲染；
4. 提示用户验证效果；
5. 若需要重启或手工操作，明确提示。

因此仅写“修改文件”通常不够；若方案包含渲染机制，就应同时写清生效链路。

---

## 3. 这些偏好为什么暂不全部写入共享规范

原因主要有三类：

1. **带有明显个人工作风格**
   - 例如更强烈地偏好最简方案、偏好内联、偏好 exact。
2. **仍依赖具体上下文**
   - 例如“默认围绕目标项目根目录设计输出”，适用于当前这类跨平台 / meta 工作流方案，但不一定适用于仓库内所有文档。
3. **还需要更多实践验证**
   - 例如某些权限策略、日志闭环、渲染节奏，虽然当前合理，但是否应升级为团队强制规则，还需要进一步观察。

---

## 4. 后续提升为共享规范的条件

若某条补充偏好连续在多个方案 / 整改文档中稳定适用，并且：

1. 不再明显依赖个人工作习惯；
2. 对团队协作有正收益；
3. 能用简洁、可执行的方式表述；

则可将其提升到：

- `AGENTS.md`（高层原则）
- `delivery/rules/AGENTS.md`（可执行细则）

---

## 5. 当前建议的使用方式

1. **写方案前**：先看共享规范；
2. **写跨平台 / 路径 / 工作流整改方案时**：再参考本文；
3. **若本文中的某条偏好已变成稳定共识**：再回写到共享规范；
4. **若本文中的某条偏好被事实证明不再适用**：直接修正文档，不保留过时表述。
