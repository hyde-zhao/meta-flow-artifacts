## 一、设计计划与推理摘要

> **注意**: design-plan.md 和 design-planner-reasoning.md 均不存在于项目仓库中，因此本节内容基于 logic-cases.md 中 LC-REL-002 的直接描述以及 P-Parameter 方法论推导。

### 1.1 设计目标

对 LC-REL-002「BGP4+ 设备级可靠性」执行 P-Parameter 判定表分析，将重启方式、配置持久化状态、操作前状态和恢复结果组合为参数化判定规则，覆盖冷重启、反复启停 50 次和配置清除恢复三种设备级可靠性场景及其异常路径。

### 1.2 分析方法

- **方法**: P-Parameter (PPDCS)，适用于「重启方式 × 配置持久化 × 恢复结果」的多因子规则判定场景。
- **CAE 聚合规则**: 规则1-参数化 —— 用规则形式表达因子间的参数化关系，而非笛卡尔积全组合。
- **判定维度来源**: 从 logic-cases.md 提取「场景(冷重启/反复启停50次/配置清除恢复) → 配置一致/丢失/残留 → 恢复结果」的核心判定链。

### 1.4 关键假设

1. 「冷重启」指设备/系统完全断电后重新加电启动，BGP4+ 配置需从持久化存储（如配置文件、NVRAM）恢复。
2. 「反复启停 50 次」指对 BGP4+ 使能（enable）进行 true/false 循环操作共 50 次，每次切换均需验证状态一致性。
3. 「配置清除恢复」指手动清除 BGP4+ 全部配置后，通过备份文件（如 frrinc.conf.bak 或类似机制）恢复并重启 bgpd。
4. 设备存储介质可靠，未发生硬件级数据损坏；若出现配置残留或丢失，归因于软件逻辑而非存储介质故障。

### 2.2 因子详细说明

#### FAC-BGP6-REBOOT-TYPE: 重启方式

- **角色**: control, driver
- **取值**: `冷重启` | `反复启停50次` | `配置清除后恢复`
- **说明**:
  - `冷重启`: 设备完全断电后重新加电启动，验证 BGP4+ 配置自动恢复及邻居重建能力。
  - `反复启停50次`: 对 BGP4+ enable 执行 true ↔ false 循环操作共 50 次，验证每次状态切换的一致性和无配置残留。
  - `配置清除后恢复`: 手动清除所有 BGP4+ 配置后，通过备份文件恢复并重启 bgpd，验证配置、邻居和路由的全面恢复。
- **驱动关系**: 该因子是判定表的主驱动轴，三种取值对应 LC-REL-002 的三个核心判定维度。

#### FAC-BGP6-CONFIG-PERSIST: 配置持久化结果

- **角色**: data, expectation
- **取值**: `一致` | `丢失` | `残留`
- **说明**:
  - `一致`: 操作前后 BGP4+ 配置完全一致，所有参数值、邻居定义、路由策略无变化。
  - `丢失`: 操作后部分或全部配置项缺失，如邻居定义消失、路由策略丢失。
  - `残留`: 操作后存在多余的、不应出现的配置项，或上一轮操作的配置未被完全清除。
- **期望关系**: 正常路径期望 `一致`；异常路径关注 `丢失` 和 `残留`。

#### FAC-BGP6-PRE-STATE: 操作前状态

- **角色**: condition
- **取值**: `Established+路由已通告+配置已保存` | `BGP4+使能` | `备份文件存在`
- **说明**:
  - `Established+路由已通告+配置已保存`: 冷重启场景的前置条件，邻居已建立、路由已通告、配置已写入持久化存储。
  - `BGP4+使能`: 反复启停场景的前置条件，BGP4+ 已启用（enable=true），准备进入循环操作。
  - `备份文件存在`: 配置清除恢复场景的前置条件，确认备份文件（如 frrinc.conf.bak）可访问且内容完整。

#### FAC-BGP6-RECOVERY-RESULT: 恢复结果

- **角色**: observation
- **取值**: `全部配置恢复+邻居重建+路由恢复` | `配置残留` | `状态异常`
- **说明**:
  - `全部配置恢复+邻居重建+路由恢复`: 完整恢复，涵盖配置加载、邻居状态重建至 Established、路由信息重新通告三个层次。
  - `配置残留`: 恢复后配置存在多余项或不一致，但不一定导致功能异常（需配合实际行为综合判断）。
  - `状态异常`: 恢复后 BGP4+ 或邻居状态异常，如邻居无法重建、路由无法恢复、进程不可用等。

## 三、Topology Binding Catalog

### 3.1 拓扑绑定说明

LC-REL-002 为设备级可靠性测试，核心关注点为单节点 DUT 的设备生命周期管理（重启、配置恢复）。不依赖复杂多节点拓扑，但需至少 1 个 BGP Peer 用于验证邻居重建和路由恢复。

| 绑定项 | 绑定内容 | 说明 |
|--------|----------|------|
| 节点类型 | 单节点 DUT（运行 FRR/bgpd） | 设备级别，关注整机生命周期 |
| 邻居拓扑 | ≥1 BGP Peer（IPv6） | 用于验证重启后邻居重建和路由恢复 |
| 配置持久化 | frrinc.conf / startup-config | 冷重启和配置恢复场景的持久化来源 |
| 备份机制 | frrinc.conf.bak 或等效机制 | 配置清除恢复场景的备份来源 |
| 控制接口 | enable=true/false（BGP4+ 使能开关） | 反复启停场景的操作入口 |

### 4.2 规则清单

#### R-REL2-01: 冷重启完整恢复（主路径）

- **条件**:
  - FAC-BGP6-REBOOT-TYPE = `冷重启`
  - FAC-BGP6-PRE-STATE = `Established+路由已通告+配置已保存`
- **预期结果**:
  - 设备重启后，BGP4+ 配置从持久化存储自动加载，与重启前一致
  - bgpd 进程正常启动，BGP 邻居自动重建至 Established
  - 路由信息恢复通告，数据面转发恢复
  - FAC-BGP6-CONFIG-PERSIST = `一致`
  - FAC-BGP6-RECOVERY-RESULT = `全部配置恢复+邻居重建+路由恢复`
- **trace_refs**: TP-Q-REL-003
- **priority**: P1
- **fact_status**: confirmed

#### R-REL2-02: 反复启停 50 次稳定性（主路径）

- **条件**:
  - FAC-BGP6-REBOOT-TYPE = `反复启停50次`
  - FAC-BGP6-PRE-STATE = `BGP4+使能`
- **预期结果**:
  - 执行 PUT enable=true ↔ enable=false 循环操作共 50 次
  - 每次 enable=true 时 bgpd 正常启动、邻居建立至 Established
  - 每次 enable=false 时 bgpd 正常停止、邻居断开、无残留进程或配置
  - 50 次循环完成后，最终状态为 enable=true、邻居 Established、配置一致
  - FAC-BGP6-CONFIG-PERSIST = `一致`
  - FAC-BGP6-RECOVERY-RESULT = `全部配置恢复+邻居重建+路由恢复`
- **trace_refs**: TP-Q-REL-004
- **priority**: P1
- **fact_status**: confirmed

#### R-REL2-03: 配置清除后完整恢复（主路径）

- **条件**:
  - FAC-BGP6-REBOOT-TYPE = `配置清除后恢复`
  - FAC-BGP6-PRE-STATE = `备份文件存在`
- **预期结果**:
  - 清除全部 BGP4+ 配置后确认配置为空
  - 通过备份文件恢复 BGP4+ 配置
  - 重启 bgpd（或等效使能操作）
  - 所有配置项完全恢复（邻居定义、路由策略、定时器参数等）
  - 邻居重建至 Established，路由信息恢复通告
  - FAC-BGP6-CONFIG-PERSIST = `一致`
  - FAC-BGP6-RECOVERY-RESULT = `全部配置恢复+邻居重建+路由恢复`
- **trace_refs**: TP-M1-023
- **priority**: P1
- **fact_status**: confirmed

#### R-REL2-04: 反复启停异常路径（异常/边界路径）

- **条件**:
  - FAC-BGP6-REBOOT-TYPE = `反复启停50次`
  - 第 N 次切换发生异常（进程未完全终止、配置写入不完整等）
- **预期结果**:
  - 第 N 次异常切换后出现配置残留（如上一次 enable=false 的中间态未清理）
  - 配置残留导致第 N+1 次 enable=true 时启动异常或状态不一致
  - **[待确认]**: 具体异常表现（进程启动失败/邻居无法建立/配置加载冲突）需结合实际实现确认
  - FAC-BGP6-CONFIG-PERSIST = `残留`
  - FAC-BGP6-RECOVERY-RESULT = `状态异常`
- **trace_refs**: 扩展（由 R-REL2-02 异常路径衍生）
- **priority**: P2
- **fact_status**: needs-confirmation
