## 治理附录（Governance）

| 字段 | 当前值 | 说明 |
|---|---|---|
| `target_artifact_type` | `agent` | 用户的直接交互入口是 `ptm-tde` Agent，由 Agent 编排 Skill 和脚本完成全流程 |

> 本附录记录"防火墙测试原子操作规范"新需求的关键场景发现结论，供后续 USE-CASES（新规范）文件和 HLD 设计直接引用。

### 三阶段调度模型（Q-02 RESOLVED）

原子操作规范的使用场景不是单点，而是跨越**三个执行阶段**的演进模型：

| 阶段 | 名称 | 执行主体 | 主要活动 | 原子操作规范的角色 |
|------|------|--------|---------|---------------|
| 阶段一 | 用例设计阶段 | 测试工程师 | 读取已有原子操作清单；为新功能设计并描述新的原子操作；输出原子操作 schema 文件 | 原子操作规范是**设计语言**，工程师按规范描述操作语义、参数和期望结果 |
| 阶段二 | 执行阶段 | 测试工程师 | 按原子操作 schema 把操作实现为 CLI 工具；人工/半自动执行；验证实现正确性 | 规范是**实现契约**，CLI 命令名称、参数列表和输出格式必须与 schema 对应 |
| 阶段三 | 自动化执行阶段 | LLM Agent | 读取 CLI 工具集 + 规范元数据；推理生成调用序列；执行端到端自动化测试 | 规范是 **LLM 可解析的调用说明书**，字段语义和机器可读格式决定调度质量 |

**关键推论**：
- 规范字段设计必须同时支持工程师阅读（自然语言描述字段）和 LLM 解析（结构化 schema 字段）。
- 原子操作从"设计描述"到"CLI 实现"再到"LLM 调度"是三个不同的产物形态，规范需要为三个阶段定义清晰的映射关系。
- 阶段三的 LLM 调度面对的是**结构化 CLI 工具集 + 规范元数据**，而非裸 REST API。

### 边界规则（Q-08 RESOLVED — 2026-05-09）

> 用户已选择候选 B 作为正式边界定义，以下规则具有约束力，后续 HLD / schema 设计必须与之对齐。

| 规则 ID | 规则内容 |
|---------|---------|
| BR-01 | **原子操作粒度**：原子操作 = 单一可观察状态变更；内部可含必要等待/重试逻辑，但不含跨目标的断言 |
| BR-02 | **测试用例结构**：测试用例 = "触发条件 → 预期状态" + 多个原子操作 + 独立断言（断言不下沉到原子操作内部） |
| BR-03 | **命名规范**：采用 `设备_动词_目标` 格式，如 `fw_ensure_acl_rule_exists`；与 Robot Framework Keyword / Ansible Task 粒度对齐 |
| BR-04 | **认可示例**：TGFW "ACL 阻断验证" = `fw_ensure_acl_rule_exists` + `tg_send_and_capture_traffic` + `fw_teardown_acl_rule` 三个原子操作 + 独立断言层 |
| BR-05 | **失败行为（Q-09/B2 RESOLVED）**：原子操作执行失败时返回结构化错误对象 `{status: "failed", error_type: "<类型>", detail: "<说明>"}`；原子操作内部不做自动重试；重试策略由编排层（测试用例执行器或 LLM Agent）控制；schema 新增必填字段 `on_failure`，枚举值为 `fail_fast / return_error / retry_by_caller`，默认 `return_error` |
| BR-06 | **幂等性约束（Q-10/I2 + Q-06 RESOLVED）**：所有原子操作默认应实现为幂等；非幂等操作必须在 schema 中显式声明 `idempotent: false`，并同时填写 `side_effects: "<副作用描述>"`；省略 `idempotent` 字段时默认视为 `true` |
| BR-07 | **跨设备编排（Q-11/E2 RESOLVED）**：原子操作始终以单设备为操作主体（`device` 字段为单值）；跨设备联动不合并为复合原子操作；测试用例 schema 新增 `tight_sequence: [op_id_list]` 字段，用于声明需要紧密顺序执行的原子操作组，不改变各原子操作本身的设计 |
