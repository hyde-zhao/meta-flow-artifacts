## 明确排除（Out of Scope）

- 使用当前仓库直接实现某个业务领域产品，而不修改 meta 工作流本身
- 运行时托管外部服务、数据库或在线编排控制台
- 脱离 `process/` / `checkpoints/` / `delivery/` 分层约束的临时流程

## 使用场景列表

### UC-01：维护者收敛和升级元工作流契约

| 字段 | 内容 |
|---|---|
| **使用角色** | 元工作流维护者 |
| **触发条件** | 需要新增或修订模式规则、阶段契约、Story / LLD 命名、Skill 输出规范或 Agent 协作边界 |
| **输入** | 当前仓库中的 `HLD.md`、`ARCHITECTURE-DECISION.md`、`STORY-BACKLOG.md`、规则文件、已有 Story / LLD 文档和用户决策 |
| **处理逻辑** | Given 当前仓库已有运行中的 meta 工作流；When 维护者发起结构性调整；Then workflow 先定位受影响的 Skill / Agent / rule / template，再统一回写正式契约文件，避免一处更新、多处滞后 |
| **输出/结果** | 更新后的 Skill、Agent、rules、模板与过程文档；必要时形成新的 Story / LLD / 变更记录 |
| **前置条件** | 关键设计文档和当前运行态文件可读取，且修改目标属于 meta-flow 自身 |
| **排除情况** | 直接为外部业务仓库产出需求、设计或实现文件，不属于当前场景 |

**处理流程（文字描述）：**
1. 读取现有 HLD、Story、rules 和交付文档，定位当前契约缺口。
2. 判断影响范围属于 Skill、Agent、模板、安装脚本还是过程文档。
3. 按单一契约源回写相关对象，并保持 `.agents/`、`delivery/`、`.github/` 口径一致。

3. 核对目标目录中的 Agent、Skill、rules 与私有资产是否完整可读。

### UC-03：维护者按 Story + LLD 增量演进 workflow

| 字段 | 内容 |
|---|---|
| **使用角色** | 元工作流维护者 |
| **触发条件** | 某个已批准 Story 需要进入 LLD、实现、验证或文档化阶段 |
| **输入** | `process/stories/STORY-*.md`、对应 `STORY-*-LLD.md`、`DEVELOPMENT-PLAN.yaml`、`TEST-STRATEGY.md`、平台规则和交付文件 |
| **处理逻辑** | Given Story 已纳入正式 backlog；When 维护者推进某个 Story；Then workflow 按 LLD 起草 → LLD 确认 → 实现 → 验证 的顺序推进，并通过 Story / LLD 文件显式交接上下文 |
| **输出/结果** | 带 `story_slug` 的 Story 卡片、LLD 文档、实现产物、验证报告与状态更新 |
| **前置条件** | HLD 已确认，Story 已批准，依赖 Story 与关键输入可读取 |
| **排除情况** | 跳过 LLD 直接实现，或在 Story / LLD 未确认时越级推进 |

**处理流程（文字描述）：**
1. 从 Story 卡片读取范围、验收标准和 `story_slug`。
2. 先产出并确认对应 LLD，再按 TASK-ID 实现和验证。
3. 回写 Story 状态、日志和交付资产，进入下一个 Wave 或阶段。

### UC-04：治理维护者扫描并整改仓库漂移问题

| 字段 | 内容 |
|---|---|
| **使用角色** | 质量与治理维护者 |
| **触发条件** | 仓库中出现路径漂移、共享模板归属错误、安装后不可执行、规则与提示词双轨、命名不直观等问题 |
| **输入** | 仓库文件树、guardrail 扫描结果、规则文件、用户反馈、相关 Story / HLD 文档 |
| **处理逻辑** | Given 当前 workflow 需要保持长期可维护；When 维护者执行审计或收到整改反馈；Then workflow 定位违规资产与契约冲突，修正规则、Skill 归属、命名、文档和安装行为，并把新约束写回正式文件 |
| **输出/结果** | 修复后的仓库结构、更新后的规则与文档、可复用的 guardrail 约束 |
| **前置条件** | 能读取当前仓库状态和相关规则；整改对象属于当前 meta 工作流 |
| **排除情况** | 只在会话中口头约定，不把约束落入正式规则或手册 |

**处理流程（文字描述）：**
1. 运行或阅读 guardrail 检查结果，识别实际漂移点。
2. 判断问题属于资产归属、命名契约、安装路径、模式规则还是过程文档缺失。
3. 修复仓库并回写规则、手册和相关 Skill / Agent 契约，防止同类问题再次出现。

### UC-05：目标负责人查看目标导向 CR 汇总

| 字段 | 内容 |
|---|---|
| **使用角色** | 审批者 / 目标负责人 |
| **触发条件** | meta-flow 因用户反馈、执行回修、CP8 follow-up 或治理改进生成多个候选 CR / 正式 CR，且用户需要判断这些 CR 对整体目标是否有价值 |
| **输入** | 用户原始目标、`process/changes/CR-INDEX.yaml`、CR summary、follow-up tracking、active / candidate CR、父子 CR 关系、冲突键和不授权项 |
| **处理逻辑** | Given 用户关心整体目标而不是内部文件分工；When meta-flow 汇总 CR 状态；Then 系统先按目标切片 / Change Package 聚合 CR，再展示每个 CR 的目标贡献、拆分理由、依赖关系、风险等级、授权边界和下一步，而不是直接要求用户逐条理解实现细节 |
| **输出/结果** | 目标导向的 CR 汇总：目标、包含 CR、为什么拆分、哪些可合并、哪些只是 follow-up、哪些需要用户决策、哪些由 agent 默认处理 |
| **前置条件** | CR summary / CR-INDEX / follow-up tracking 可读取；至少存在一个 active、candidate 或 proposed CR |
| **排除情况** | 把每个实现任务、文件变更或低风险命名选择都提升为独立正式 CR |

**处理流程（文字描述）：**
1. 读取 CR index、summary 和 follow-up tracking，按 `goal_ref`、`parent_cr`、`conflict_keys` 或人工指定目标聚合。
2. 对每个 CR 标注“服务的整体目标、拆分理由、是否必须独立审批、是否可降级为 Story / task / follow-up”。
3. 输出一页目标导向摘要，完整 CR 文件只作为可展开证据。

### UC-06：审批者处理带影响摘要的人工确认

| 字段 | 内容 |
|---|---|
| **使用角色** | 审批者 / 目标负责人 |
| **触发条件** | meta-flow 发起 CP2 / CP3 / CP5 / CP8 人工确认，或某个 CR 需要用户 approve / 修改 / reject |
| **输入** | Decision Brief、Context Capsule、自动预检结果、待人工决策清单、不授权项、风险与回退信息 |
| **处理逻辑** | Given 审批者不应先阅读完整 CR / LLD 才能理解影响；When meta-flow 发起人工确认；Then 确认消息先展示本次确认服务的整体目标、推荐动作、approve 后会发生什么、approve 不授权什么、不确认会阻塞什么，再展示必须用户决策项和完整 evidence 路径 |
| **输出/结果** | 可直接决策的人工确认摘要，支持 `approve`、`修改: <具体修改点>`、`reject`；低风险细节不进入主决策表 |
| **前置条件** | 自动预检已完成；Decision Brief 已收集候选问题并完成用户决策分层 |
| **排除情况** | 只输出技术决策表，不说明整体目标、用户影响、授权边界和不确认后果 |

**处理流程（文字描述）：**
1. 聚合 Decision Brief、预检结论和 pending decisions。
2. 将决策项分层为“必须用户决策 / 高风险策略确认 / agent 默认处理 / 仅审计记录”。
3. 向用户展示审批者摘要和精简决策表，保留完整 checkpoint 路径供审计展开。

### UC-07：维护者通过风险分级降低确认噪音

| 字段 | 内容 |
|---|---|
| **使用角色** | 质量与治理维护者 |
| **触发条件** | 人工确认中出现大量可逆、局部、低风险实现取舍，例如命令命名、内部文件格式主写、文档措辞、局部测试组织方式 |
| **输入** | gate profile、风险规则、决策候选、可逆性判断、用户可见性、公共契约影响、安全 / 运行授权边界 |
| **处理逻辑** | Given 不是所有实现取舍都需要用户拍板；When meta-flow 收集到候选决策；Then 系统按风险、可逆性、用户可见性、是否改变公共契约和是否涉及授权边界分级，只把范围、风险、授权、不可逆或用户可见行为变化推给用户确认，其他由 agent 采用推荐默认并留审计摘要 |
| **输出/结果** | 更少、更高价值的人工确认项；完整默认处理清单作为审计附录保留 |
| **前置条件** | gate profile 和风险规则可用；Decision Brief 支持记录 `decision_owner`、`blocking_level`、`reversibility`、`user_visible` 与 `default_action` |
| **排除情况** | 将 runtime、凭据、外部写入、publish、交易、生产写入等高风险事项降级为 agent 默认处理 |

**处理流程（文字描述）：**
1. 对候选决策运行风险分级，识别用户必须确认的范围、架构、安全、运行授权和风险接受项。
2. 对低风险实现细节记录推荐默认方案、理由和回退条件，不进入用户主确认表。
3. 在 CP8 或对应检查点中汇总“agent 默认处理项数量、例外项、审计路径”。

### UC-08：维护者对新增需求做入口分流

| 字段 | 内容 |
|---|---|
| **使用角色** | 元工作流维护者、审批者 / 目标负责人 |
| **触发条件** | 用户提出新增需求、补充需求、修改范围、启动 follow-up 候选、提交一组集中需求、或要求推进某个目标包时 |
| **输入** | 用户本轮原始输入、当前 `STATE.current.json` / `STATE.md`、active CR / CR summary / CR-INDEX、follow-up tracking、既有 `USE-CASES.md` / `REQUIREMENTS.md` / HLD / Story backlog、目标项目交付约定、不授权边界 |
| **处理逻辑** | Given 用户输入可能是大块集中需求、单点变更、零散后续、运行授权或设计澄清；When meta-flow 接收到本轮请求；Then Host Orchestrator 先输出入口分流判断，说明推荐路径、判定依据、备选路径、会不会创建 CR、是否需要完整 CP2 / CP3 / Story 拆分，并请求用户确认或允许 agent 采用默认分流 |
| **输出/结果** | 分流决策摘要；必要时创建一个 parent CR / Change Package 作为审计外壳；大块需求进入 `requirement-clarification -> CP2 -> solution-design -> CP3 -> story-planning`；单点变更进入精准 CR；零散后续进入 follow-up tracking；运行授权进入 runtime authorization 门禁；低风险实现细节进入 agent 默认处理并留痕 |
| **前置条件** | process 路由健康；可读取当前状态和 CR 跟踪摘要；若要修改正式产品基线，必须已有或同步创建 CR 审计外壳与文档处理决策 |
| **排除情况** | 在未完成分流前逐条创建正式 CR；把同一目标下的多个需求点拆成多个 CR；在 CP2 / CP3 未通过前直接进入 Story / LLD / 实现；把运行授权、安全接受或生产写入混入普通产品需求确认 |

**分流原则：**

| 分流类型 | 判定条件 | 默认路径 | 用户交互模式 | 不应做的事 |
|---|---|---|---|---|
| 大块集中需求 / 目标包 | 同一用户目标下有多个需求点；影响多个 use case / feature / story；会改变用户旅程、成功指标、范围边界或架构取舍；需要 HLD 或多个 Story 才能交付 | 创建一个 parent CR 或 Change Package 作为审计外壳，然后进入完整 `requirement-clarification -> CP2 -> solution-design -> CP3 -> story-planning` | 向用户展示“这是一个目标包”的推荐判断，列出 2-3 个候选范围和 trade-off；每轮只问一个会改变范围 / HLD / Story Map 的问题；用户确认后再委托 meta-pm | 不逐条拆 CR；不直接做 Story / LLD；不把 CR 当成需求组织单元 |
| 单点产品变更 | 只影响一个明确需求或一个局部行为；不需要新增 HLD；验收边界清晰；风险和授权边界单一 | 创建单个 CR，完成五维影响分析和必要的增量需求 / 场景更新；若影响产品基线仍回 CP2 | 输出简短 CR 摘要、影响面、是否需要 CP2；用户可回复 `approve` / `修改: ...` / `reject` | 不升级为完整工作流；不拆成多个 CR |
| 零散后续事项 | 来源于 CP8、反馈、观察项或 deferred idea；尚未决定推进；与当前目标包不一定同构 | 进入 follow-up tracking；用户决定推进时再做冲突预检；多个同目标候选先合并为目标包 | 展示候选项、优先级、阻塞前置、合并建议和不授权边界；用户选择启动、合并、延期、取消或转 Spike | 不预创建正式 CR；不把候选列表直接当开发计划 |
| 设计澄清 / 返工 | CP7 `NEEDS_DESIGN_CLARIFICATION`、HLD / LLD 发现边界冲突、Story 范围不稳定 | 回到 CP3 / CP5 或创建设计澄清 CR；只在产品基线变化时回 CP2 | 向用户说明冲突点、推荐回退阶段、影响 Story 和风险；请求确认回退或 Spike | 不在实现阶段临时扩大 Story 范围 |
| 运行授权 / 安全接受 | 涉及凭据、外部服务、生产写入、publish、live / trading、真实运行或风险接受 | 单独形成 `runtime_authorization` / `security` 决策项或授权门禁，不混入普通 approve | 明确列出“approve 不授权”的事项；需要真实执行时单独请求授权，说明命令、环境、回滚和风险 | 不把 CP2 / CP5 / CP8 设计确认解释为运行授权 |
| 低风险实现细节 | 可逆、局部、非用户可见、不改变公共契约、不涉及安全 / 运行授权 | 由 agent 默认处理，写入审计摘要或实现说明 | 在人工门禁中汇总数量、默认策略和例外项；只把高风险项推给用户 | 不升级为用户必须决策项；不创建独立 CR |

**用户交互模式：**

1. **先给分流结论，再问问题。** Host Orchestrator 收到新增需求后，先输出一段可审查的分流摘要：`我判断这是 <分流类型>，推荐路径是 <路径>，原因是 <依据>，不会授权 <不授权项>`。
2. **大块需求只问目标包级问题。** 对大块集中需求，优先询问目标、用户角色、成功指标、范围边界、优先级和关键取舍；不在入口阶段逐条追问每个实现细节。
3. **一次只问一个高价值问题。** 每个问题必须能改变范围、HLD、Story Map、风险接受或交付路线；普通字段补齐放到后台扫描或 `REQUIRED` 默认假设。
4. **提供候选路径和 trade-off。** 至少给出推荐路径和一个备选路径，例如“完整产品流程”“单个 CR”“follow-up 延后 / Spike”，并说明对时间、风险、文档、门禁和开发的影响。
5. **用户可自由纠正。** 如果用户否认推荐分流，系统必须复述新的理解并更新分流类型，不继续强推原选项。
6. **确认语义必须精确。** 对分流确认只使用 `approve`、`修改: <具体修改点>`、`reject`；`approve` 表示接受推荐分流和后续门禁路径，不表示授权真实运行、读取凭据、生产写入或跳过 CP2 / CP3 / CP5 / CP8。
7. **分流结果必须落盘。** 分流判断写入 discussion log、CR summary / Change Package summary 或 checkpoint 输入；后续 agent 只消费落盘结果，不依赖对话隐式记忆。

**处理流程（文字描述）：**
1. 读取当前状态、CR 跟踪摘要、follow-up 台账和既有产品 / 设计基线，确认是否已有 active CR 或待处理目标包。
2. 对用户本轮输入执行入口分类，判断是大块集中需求、单点产品变更、零散后续、设计澄清、运行授权还是低风险实现细节。
3. 若命中大块集中需求，创建或复用一个 parent CR / Change Package 作为审计外壳，记录旧基线保留方式和产品基线刷新要求。
4. 委托 meta-pm 执行完整用户场景发现、需求澄清、SCENARIOS / TEST-MATRIX / STORY-MAP / MVP-SCOPE，并通过 CP2。
5. CP2 通过后委托 meta-se 执行 HLD / Architecture Gray Areas / ADR，CP3 通过后再进行 Story 拆分和开发计划。
6. 仅当 HLD 或门禁证明目标、审批人、风险授权、交付节奏或回滚策略不同，才拆出子 CR；普通开发工作拆 Story。
