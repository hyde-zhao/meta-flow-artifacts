## 变更描述

用户要求优化 meta-flow 工作流：整个流程中所有需要人工确定的信息，都必须提供推荐方案、至少 1 个备选方案（优先 2 个）和优劣分析；meta-po 发起人工确认时，必须收集所有待人工确认的问题、推荐方案和备选方案，并打印给用户统一决策。

## 影响分析

| 维度 | 结论 |
|---|---|
| 需求层 | 人工确认协议从“确认产物”升级为“逐项决策清单 + 推荐 / 备选 / 优劣”。 |
| 场景层 | CP2 / CP3 / CP5 / CP8 以及 fast-lane brief 都需要汇总待人工决策项。 |
| 计划层 | 不改变 CP0-CP8 阶段和门禁顺序；只增强 Decision Brief 和 clarification queue 的字段契约。 |
| 安全层 | inline fallback、风险接受和 WAIVED 项必须作为决策项暴露，降低隐式风险接受。 |
| 交付层 | 同步更新交付 Agent、Skill、rules、README 和 USER-MANUAL，避免安装后平台规则分叉。 |

## 决策

| 决策点 | 结论 |
|---|---|
| 全局人工决策结构 | 所有需要人工确定的信息都形成待人工决策项 |
| 最小备选数量 | 每项至少 1 个可执行备选，优先 2 个 |
| 无业务备选时的处理 | 不写“无备选”，改给治理备选：暂缓确认、保持当前基线、缩小范围、回退上游或转 Spike |
| Codex exact 文本协议 | 仍只展示 `approve`、`修改: <具体修改点>`、`reject`；`approve` 表示接受清单内全部推荐方案 |
| Claude Code 结构化选择 | 保留结构化选择能力，但仍必须先展示同等信息量的决策项 |

## 风险与回退

| 风险 | 影响 | 缓解 / 回退 |
|---|---|---|
| 决策清单过长 | 用户确认负担增加 | 只汇总影响阶段推进、范围、架构、文件所有权、风险接受、交付出口的决策；细节留在 checklist |
| 平台规则分叉 | Codex / Claude 行为不一致 | 同步 delivery rules、agent 源和当前运行副本，并运行 guardrail |
| 误把阶段内探索变成长问卷 | 降低澄清效率 | 保留阶段内一次一问；仅正式人工检查点前由 meta-po 批量汇总 |
