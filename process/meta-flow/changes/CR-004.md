## 变更描述

用户提出 8 项 meta-flow 工作流改进，目标是降低 Codex 子 agent 数量与 token 消耗，明确子 agent 生命周期，优化安装器为 `uv tool` 全局命令，引入更强的 use-case 头脑风暴，合并 Story 计划确认与 LLD 确认，增加平台化人工确认协议，并防止 production 项目误写当前仓库 `delivery/`。

变更范围：

1. `meta-po` 子 agent 单例化：同一工作流只允许 1 个 `meta-po`，由该 `meta-po` 调度其他 meta 子 agent。
2. Codex 子 agent 上下文精简与 token 控制：显式 context handoff、最小上下文包、低 fan-out、不 fork 全量上下文、配置建议。
3. 子 agent registry、复用与确认后关闭：同一任务 / Story 复用同一线程，确认点完成后及时 close。
4. 安装器优化：新增 `uv tool` 全局命令，CLI 建议 `scope-pack install`；组件改为 `rules|agent|full`；user scope 默认 `rules`，project scope 默认 `agent`；保留 legacy `--content`。
5. use-case 头脑风暴增强：参考 superpowers 的先发散、一次一问、2-3 方案比选、分段确认，不照搬外部实现。
6. 流程调整：`meta-se` 完成 Story 分解后，组织 `meta-dev` 完成 LLD 设计；Story 确认和 LLD 确认合并为 Story Package 确认，优先按 wave 合并。
7. 平台化人工确认协议：Claude Code 保留结构化选择；Codex 也优先支持结构化选择；若当前 Codex 客户端 / 模式无法提供，则显式降级为 exact 文本确认，文本仅作为兜底。
8. 交付件目录路由：只有 meta-flow 自身改进 / `meta-self-dev` 才直接写当前仓库 `delivery/`；其他 production 项目必须先扫描目标 README / docs 的交付物约定，存在则遵守，不存在则提出建议并询问用户确认后再写。

## 子 agent 生命周期计划

当前会话无法实际控制 Codex 下游 agent 线程，因此本 CR 不声称已拉起或关闭任何下游子 agent。实施时由唯一 `meta-po` 主线程按下列计划调度：

| 角色 | 最大实例数 | 复用键 | 关闭时机 |
|---|---:|---|---|
| `meta-po` | 1 | workflow_id | 工作流 delivered 或人工明确终止后关闭；不得由 `meta-po` 再拉起 `meta-po` |
| `meta-pm` | 1 | workflow_id + change_id | USE-CASES / REQUIREMENTS 或 CR 场景分析交付后关闭 |
| `meta-se` | 1 | workflow_id + change_id | HLD/ADR/Story Package 计划交付并通过 review gate 后关闭 |
| `meta-dev` | 每个 active Story / wave 最多 1 个 | story_id 或 wave_id | LLD 包交付后暂停；Story Package 确认通过后复用同线程实现；实现完成交接 meta-qa 后关闭 |
| `meta-qa` | 每个 active Story / wave 最多 1 个 | story_id 或 wave_id | 验证报告和安装回归结果交付后关闭 |
| `meta-doc` | 1 | workflow_id + doc_phase | README / USER-MANUAL 修订完成并交付终验后关闭 |
