## 变更描述

用户要求补齐 meta-flow 的验证环节，不只验证 meta-flow 自身，还要验证 meta-flow 生产的其他工作流；同时 meta-flow 也会开发纯代码项目，因此验证机制不能破坏纯代码项目的既有验证路径。用户进一步要求采用稳定、工程化的方法完成 meta-flow 治理，并先完成 CR 写作。

本 CR 建立治理基线，不在本 CR 内直接实现 runner、grader、CI 或外部平台集成。核心决策如下：

1. 不把 meta-flow 拆成“纯代码开发工作流”和“工作流开发工作流”两个完整主流程；保留单一主流程，增加项目画像和 Story 级验证目标进行分流。
2. 保留既有 `engagement_mode` 职责，只用于区分 `meta-self-dev` 与 `production` 交付路由，不承载项目类型判断。
3. 新增项目画像：`target_project_profile.project_kind = code-project|workflow-product|agentic-code-product|mixed|unknown`。
4. 新增 Story / 交付对象级验证目标：`validation_target.sut_type = code-project|generated-workflow|prompt-skill-workflow|meta-flow-core-code|agentic-code-product|mixed`。
5. 将 workflow eval 和 prompt bundle 作为一等治理对象：
   - workflow eval 负责验证生成工作流的结构、路由、门禁、追踪、dry-run、权限边界、故障恢复和回归稳定性；
   - prompt bundle 负责固化规则、Agent、Skill、状态路由、context policy、human gate 文案、评估提示词和 rubric 的版本、hash、变更类型与兼容性。
6. 对纯代码项目保持条件触发：`project_kind=code-project` 且 Story `sut_type=code-project` 时，默认执行目标项目原生测试、静态检查、构建、质量评审和发布检查；不强制引入 workflow eval。
7. 对 workflow product / agentic-code-product / mixed 项目增加分层验证：
   - 代码层：单元测试、集成测试、静态检查、构建和目标项目原生验证；
   - 工作流层：结构契约、DAG/状态机、human gate、handoff、checkpoint、trace、dry-run、权限和失败恢复；
   - prompt 层：prompt bundle hash、rubric、golden case、negative case、regression case、multi-run 稳定性；
   - 交付层：安装包、平台目录、文档、发布/回滚、目标项目路由。
8. 按研究文档中的方法采用 trace-first、dataset-second、CI-third 的渐进策略：先统一 trace / 证据 schema，再补 case registry / fixtures / graders，最后接入 CI 和可选外部工具。
